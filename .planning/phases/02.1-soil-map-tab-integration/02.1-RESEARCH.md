# Phase 2.1: Soil Map Tab Integration - Research

**Researched:** 2026-04-30
**Domain:** React Leaflet vector overlays, static asset sync, lazy GeoJSON loading
**Confidence:** HIGH (all key findings verified against the live repo state)

---

<phase_scope>
## Scope Summary

Phase 2.1 bridges the completed BUEK250 vector pipeline into the frontend. The pipeline already produces committed per-LL GeoJSON fixtures in `data/geojson/`, but the app does not yet publish or render them. The missing work is:

- sync the per-LL GeoJSON files into `app/public/data/` so the static app can fetch them
- expose a frontend layer contract for a vector-backed soil tab
- render the soil overlay lazily inside the existing `LLMap` experience
- keep non-soil layers and the rest of the LL detail shell working if the soil layer is loading or fails

This phase should not change the Phase 2 build contract (`data/geojson/buek250-{slug}.geojson` stays the source artifact pattern).
</phase_scope>

<repo_findings>
## Verified Repo Findings

### 1. The pipeline output exists, but the app cannot fetch it yet

- `data/geojson/` contains five committed files:
  - `buek250-east-brandenburg.geojson`
  - `buek250-havellandisches-luch.geojson`
  - `buek250-hessian-low-mountain.geojson`
  - `buek250-north-hessian-loess.geojson`
  - `buek250-rheingau.geojson`
- `data-pipeline/sync.py` copies metadata, LL boundary GeoJSON, and PMTiles into `app/public/data/`, but it has no vector-GeoJSON sync step.
- Result: the browser cannot currently request `data/geojson/buek250-{slug}.geojson` from the built app.

### 2. The map shell is already a strong fit for incremental vector support

- `app/src/pages/LLDetail.jsx` lazy-loads `LLMap` and switches layers with `startTransition`.
- `app/src/components/LLMap/index.jsx` already:
  - loads the selected LL boundary with `useGeoJSON('data/ll_boundaries.geojson')`
  - computes map bounds and the white outside-mask
  - shows loading/error fallback UI
  - keeps legend and source-attribution controls inside the map shell
- This means the soil layer can be added as an overlay inside the existing map instead of creating a separate soil-specific page flow.

### 3. Current layer config only supports raster overlays

- `app/src/data/layers.js` marks `soil` as `available: false` and only models raster overlays via `pmtilesUrl`.
- `LLMap/index.jsx` only mounts overlay content through `RasterPmtilesLayer`, which assumes PMTiles raster data.
- A vector-capable layer contract is needed before the soil tab can be enabled.

### 4. The existing `useGeoJSON` hook already provides the right lazy-loading primitive

- `app/src/hooks/useGeoJSON.js` caches responses by URL, deduplicates in-flight requests, and supports one or more URLs.
- It is already used for the boundary file and can be reused for the soil GeoJSON request keyed by the active LL slug.
- Because the hook fetches only when mounted, the cleanest lazy-load strategy is to mount the soil overlay request only when `layer === 'soil'`.

### 5. Source attribution support is present but stale for the new soil layer

- `app/src/data/layer_sources.js` is generated from `data-pipeline/sources/sources.yaml`.
- `sources.yaml` already contains a `buek250` layer with `app_layer: soil`.
- The generated file currently lists only the landuse layer, which means `sync.py` has not been rerun since the soil source was declared.
- Once `sync.py` publishes vector assets and regenerates `layer_sources.js`, the map info control can show correct BGR attribution for the soil tab without extra frontend hardcoding.

### 6. The committed BUEK GeoJSON supports a practical first-pass styling strategy

- Soil polygons carry `SYM_NR` and `GEN_ID`.
- Many water or non-soil features carry `BEMERKUNG` with `SYM_NR`/`GEN_ID` null.
- Example from `buek250-east-brandenburg.geojson`:
  - total features: 572
  - features with `SYM_NR`: 176
  - features with `BEMERKUNG`: 396
- This supports a deterministic first-pass style:
  - water/special-area polygons: one muted water style
  - soil polygons: deterministic fill chosen from `SYM_NR`
- A perfect reproduction of the desktop QGIS styling is not required for this phase to meet the roadmap goal of “stable styling.”

### 7. Current legend/UI scaffolding can absorb a first shipping version

- `MapLegend.jsx` already supports either:
  - a generated legend array attached to the layer config, or
  - a simple fallback category palette from `LAYER_COLORS`
- `i18n.js` already contains `layers.soil`, `legend.soil.*`, and the generic map loading/error strings.
- This means the phase can ship with a modest, deterministic soil legend rather than waiting for a full soil-class taxonomy export.
</repo_findings>

<recommended_approach>
## Recommended Implementation Approach

### A. Publish the vector assets through the existing sync path

Keep Phase 2 outputs in `data/geojson/` as the source of truth. Extend `data-pipeline/sync.py` to copy `buek250-{slug}.geojson` files into `app/public/data/geojson/` during the normal sync run. This keeps the app static and preserves the existing offline pipeline model.

### B. Add a vector layer contract instead of special-casing soil everywhere

Extend `app/src/data/layers.js` with enough metadata to describe a vector-backed layer. Recommended shape:

- `type: 'raster' | 'vector'`
- `pmtilesUrl` for raster layers
- `geojsonPathPattern` for vector layers, using `{slug}` substitution
- `available`
- optional legend/style metadata if needed later

This keeps `LLMap` generic and makes future vector layers cheaper.

### C. Keep overlay loading independent from base map and boundary rendering

Do not reuse the current top-level `loading` / `error` branches for the soil overlay request. Those branches currently gate the entire map shell based on boundary availability. Instead:

- keep the boundary fetch as the shell prerequisite
- fetch the soil GeoJSON only when `layer === 'soil'`
- if soil is loading, keep the map shell visible and show an overlay-level loading state
- if soil fails, keep the LL outline/basemap visible and show an overlay-level error message

This directly satisfies the roadmap requirement that the vector layer load not block the rest of the LL detail page.

### D. Start with deterministic styling, not a full semantic soil legend

Recommended first shipping rule set:

- `BEMERKUNG` present and `SYM_NR` missing -> water/special-area style
- otherwise style by deterministic color mapping from `SYM_NR`
- keep stroke subtle so the boundary outline remains readable

This delivers “stable styling” now while leaving room for a future richer legend or profile lookup keyed by `GEN_ID`.
</recommended_approach>

<pitfalls>
## Likely Pitfalls

### Pitfall 1: enabling the soil tab before the runtime asset exists
- If `layers.soil.available` is flipped to `true` before `sync.py` copies the GeoJSON into `app/public/data/geojson/`, the browser will hit 404s.
- Fix: make asset publication part of the same plan wave as the runtime contract and verify with `python data-pipeline/sync.py`.

### Pitfall 2: using one loading flag for both boundaries and overlay data
- The current `LLMap` early returns are correct for shell prerequisites, but using them for soil overlay loading would blank the whole map.
- Fix: split “boundary shell ready” from “soil overlay ready.”

### Pitfall 3: reusing the raster overlay path for vector data
- `RasterPmtilesLayer` is PMTiles-specific. Trying to wedge GeoJSON into that path will produce brittle branching.
- Fix: add a separate vector overlay path, ideally with a small `SoilVectorLayer` component.

### Pitfall 4: overcommitting to a perfect soil legend in this phase
- The repo has enough data to render stable polygons now, but not enough app-side infrastructure to explain every BUEK class semantically.
- Fix: ship deterministic map styling now; leave full semantic chart/legend work to later phases.
</pitfalls>
