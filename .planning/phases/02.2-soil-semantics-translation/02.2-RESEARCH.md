# Phase 2.2: Soil Semantics & Translation - Research

**Researched:** 2026-04-30
**Domain:** BUEK250 SQLite semantics, translation/normalization strategy, frontend soil contract migration
**Confidence:** HIGH (repo state, SQLite schema, sample records, and BGR XML metadata verified locally)

<phase_scope>
## Scope Summary

Phase 2.2 stabilizes the soil metadata contract emitted by the BUEK250 vector pipeline before the app builds more UI on top of it.

The current export path already joins the SQLite database, but it stops at a shallow lookup:

- `build_vector.py` merges only `buek250_Legendeneinheit__v10_tbl`
- emitted GeoJSON exposes `soil_name`, `soil_type_1`, `soil_type_2`, `soil_type_3`
- those fields are German-first, partially sparse, and not explicitly modeled for provenance or translation
- the frontend still styles soil polygons from raw fields like `SYM_NR` and `BEMERKUNG`

The missing work is therefore not “add any lookup”, but “replace the weak lookup with a documented bilingual contract that reflects the actual BUEK250 database structure and that the frontend can trust.”
</phase_scope>

<verified_findings>
## Verified Findings

### 1. The current pipeline already proves the integration seam, but not the final contract

- `data-pipeline/python/build_vector.py` reads the BUEK250 GeoPackage and merges a lookup frame from SQLite.
- The lookup is currently defined in `data-pipeline/sources/sources.yaml` under `vector.lookup`.
- That lookup only selects:
  - `LE_TXT -> soil_name`
  - `BOD_FT1 -> soil_type_1`
  - `BOD_FT2 -> soil_type_2`
  - `BOD_FT3 -> soil_type_3`
- `data-pipeline/tests/test_pipeline_outputs.py` currently locks those exact weak fields into the contract.

Implication: Phase 2.2 must change both the export and the tests. Otherwise the old field names will keep the weak contract alive indefinitely.

### 2. The BGR metadata explicitly says the shallow legend text is only part of the story

The downloaded BGR XML metadata for the SQLite database states:

- `LEGENDENEINHEIT` contains the textual descriptions of legend units
- `PROFIL` and `HORIZONT` drive thematic evaluations
- `GL_EINHEIT` and `GL_BAGFlaechentyp` support higher-level grouping by general legend units and parent-material surface types

This aligns with the current roadmap wording and gives a clear modeling direction:

- `LEGENDENEINHEIT` is suitable for display labels and direct polygon lookup
- `PROFIL` / `HORIZONT` are suitable for richer semantic summaries
- `GL_EINHEIT` / `GL_BAGFlaechentyp` are suitable for grouped classifications that are easier to translate and explain in the UI

### 3. The relevant SQLite tables and key join fields are straightforward

Verified table names:

- `buek250_Legendeneinheit__v10_tbl`
- `buek250_Profil__v10_tbl`
- `buek250_Horizont__v10_tbl`
- `buek250_GL_Einheit_v10_tbl`
- `buek250_GL_BAGFlaechentyp_v10_tbl`

Verified join anchors:

- `GEN_ID` links polygons to `LEGENDENEINHEIT`
- `LEGENDENEINHEIT.GL_ID` links to `GL_Einheit`
- `LEGENDENEINHEIT.GEN_ID` links to one or more `PROFIL` rows
- `PROFIL.BF_ID` links to one or more `HORIZONT` rows
- `GL_Einheit.BAG_FT_ID` links to `GL_BAGFlaechentyp`

Implication: the new contract can be built deterministically from repo-local data without calling external services or inventing opaque heuristics.

### 4. The raw text really is not UI-ready yet

Sample records confirm the current problems:

- `LE_TXT` is descriptive but German-only and often long
- `BOD_FT3` is frequently null
- profile and horizon tables contain many abbreviated coded fields that need curation before becoming runtime data
- emitted GeoJSON already shows `BEMERKUNG`-only special areas mixed with soil polygons, so the contract must distinguish semantic soil units from special/water areas explicitly instead of relying on null checks downstream

Implication: Phase 2.2 should not just rename the current fields. It needs a normalized nested structure with:

- explicit category or feature kind
- bilingual display labels
- preserved German source values where traceability matters
- intentional null handling

### 5. The frontend still depends on pre-semantic cues

Verified frontend usage:

- `app/src/components/LLMap/index.jsx` styles ordinary polygons from `SYM_NR` / `GEN_ID`
- special areas are inferred from `BEMERKUNG` combined with missing `SYM_NR`
- `app/src/data/layers.js` still presents a generic two-item soil legend
- there is no stable app-facing semantic object yet, only ad hoc field inspection

Implication: the frontend migration plan should replace direct dependence on `SYM_NR`, `BEMERKUNG`, and `soil_type_*` as the primary UI contract, while still retaining low-level identifiers for traceability and debugging.
</verified_findings>

```json
    },
    "labels": {
      "de": {

      },
      "en": {
        "legend_unit": "...",

    },
    "classifications": {
      "soil_types": [
        { "de": "Braunerden", "en": "brown soils", "source_field": "BOD_FT1" }
      ],
      "profile_summary": {
        "dominant_profile_de": "...",

      "parent_material": {
        "code": "1.0.1",

        "label_en": "marine sediments"
      }
    },
    "provenance": {
      "legend_table": "buek250_Legendeneinheit__v10_tbl",
      "profile_table": "buek250_Profil__v10_tbl",
      "horizon_table": "buek250_Horizont__v10_tbl",
      "general_legend_table": "buek250_GL_Einheit_v10_tbl",

    }
  },
  "source_note_de": "...",
  "source_note_en": "..."

```

Important implementation note:

- the final field names can be flatter than this if file size becomes a concern
- the contract still needs these concepts, even if the JSON encoding is more compact

Recommended migration rule:

- keep `GEN_ID`, `SYM_NR`, and `BEMERKUNG` for provenance and debugability
- stop treating `soil_name` and `soil_type_*` as the public UI contract
- emit a new normalized semantic object or flattened semantic namespace that the app uses first
</recommended_contract>

<translation_strategy>
## Translation Strategy

The English layer should be reproducible and committed in the repo, not generated manually inside GeoJSON outputs.

Recommended approach:

- create a dedicated repo-tracked translation/normalization asset for recurring German source terms
- translate stable categorical vocabularies first:
  - `BOD_FT1..3`
  - `BO_SUBTYP_TXT`
  - `BODSYSTEINH_TXT`
  - `GL_TXT`
  - `BAG_FT_TXTK`
- keep long free-text legend descriptions traceable:
  - cleaned German source string
  - deterministic English translation produced from a committed mapping or transformation layer

Do not:

- hand-edit translated strings inside emitted GeoJSON fixtures
- hide provenance when a label is derived, shortened, or translated
- expose raw malformed strings when they can be normalized during build time
</translation_strategy>

<implementation_guidance>
## Recommended Implementation Split

### Wave 1: Define the semantic contract and translation/cleanup assets

- introduce a dedicated helper module for soil-semantic extraction and normalization
- document the new app-facing fields in `data-pipeline/README.md`
- move translation and normalization data into a committed source file rather than embedding it ad hoc in `build_vector.py`

### Wave 2: Emit the richer contract and refresh tests/fixtures

- replace the current `soil_name` / `soil_type_*`-only export path
- join `LEGENDENEINHEIT`, `GL_Einheit`, and `GL_BAGFlaechentyp`
- derive a compact profile summary from `PROFIL` and, if needed, `HORIZONT`
- rebuild and recommit the five `data/geojson/buek250-*.geojson` fixtures
- update `pytest` assertions to validate the new semantic contract instead of the old weak fields

### Wave 3: Migrate frontend usage to the new contract

- update map styling and legend/info logic to read semantic fields first
- stop treating null checks on raw fields as the main soil UI behavior
- preserve resilience: special/water areas should still render cleanly without blanking the map shell
</implementation_guidance>

<pitfalls>
## Likely Pitfalls

### Pitfall 1: preserving old fields as the primary contract “for compatibility”

If the frontend and tests continue to rely on `soil_name` / `soil_type_*`, the new contract will exist only on paper.

### Pitfall 2: pulling too much raw `HORIZONT` detail directly into GeoJSON

`HORIZONT` is rich but verbose and coded. Dumping raw horizon rows into every polygon feature would bloat files and create a brittle UI contract.

Recommended fix: derive a compact summary for the runtime asset and keep the deeper SQLite tables as the build-time authority.

### Pitfall 3: mixing display labels with provenance-less translated strings

If translated labels overwrite the original German source values, debugging or future retranslation becomes harder.

Recommended fix: keep source and translated values separate wherever the translation is non-trivial.

### Pitfall 4: leaving special areas defined only by missing soil IDs

The current “`BEMERKUNG` and null `SYM_NR`” inference works, but Phase 2.2 should turn that into an explicit exported category so the frontend does not have to reverse-engineer intent.
</pitfalls>
